    S-01：academic-paper-review（学术论文评审）¶

    Frontmatter 描述译注：

    当用户请求评审、分析、批评或摘要学术论文、研究文章、预印本或科学出版物时使用此技能。支持覆盖方法论评估、贡献评价、文献定位和建设性反馈生成的全面结构化评审。在涉及论文 URL、上传 PDF、arXiv 链接，或请求"评审这篇论文"、"分析这项研究"、"总结这项研究"或"写一份同行评审"的查询上触发。

    概述¶

    本技能对学术论文和研究出版物进行结构化的同行评审质量分析。遵循顶级期刊（NeurIPS、ICML、ACL、Nature、IEEE）使用的既定学术评审标准，提供严谨、建设性和均衡的评估。

    评审涵盖摘要、优势、不足、方法论评估、贡献评价、文献定位和可操作建议——所有内容都基于论文本身的证据。

    何时使用¶

    • 用户提供论文 URL（arXiv、DOI、会议论文集、期刊链接）
    • 用户上传研究论文或预印本的 PDF
    • 用户请求"评审"、"分析"、"批评"、"评估"或"摘要"研究论文
    • 用户想了解研究的优势和不足
    • 用户请求学术评审风格的评估
    • 用户寻求为会议或期刊投稿准备评审

    核心工作流¶

    第一阶段：理解论文

    • 步骤 1.1：提取并记录论文元数据（标题、作者、发表地点/状态、年份、领域、论文类型）
    • 步骤 1.2：系统性深读——摘要与引言 → 相关工作 → 方法论 → 实验/结果 → 讨论与局限性 → 结论
    • 步骤 1.3：明确列出主要声明，标注每项声明的证据和强度（强/中/弱）

    第二阶段：批判性分析

    • 步骤 2.1：使用网络搜索了解研究背景
    • 步骤 2.2：使用 6 维评估框架对方法论评分（合理性、新颖性、可重现性、实验设计、统计严谨性、可扩展性），每项 1-5 分
    • 步骤 2.3：按 5 级贡献意义评估（里程碑式 / 重大 / 适中 / 边际 / 未达标）
    • 步骤 2.4：为每个优势和不足提供：是什么（具体观察）、在哪里（章节/图/表引用）、为何重要（对论文主张或效用的影响）

    第三阶段：综合评审

    • 步骤 3.1：使用规定模板组装完整结构化评审

    评审输出模板关键结构¶

    # 论文评审：[论文标题]
## 论文元数据
## 执行摘要      ← 2-3 段，提前说明总体判断
## 贡献摘要
## 优势          ← 每项带具体引用
## 不足          ← 每项带修复建议
## 方法论评估    ← 6 维评分表
## 作者提问
## 小问题
## 文献定位
## 建议          ← 总体判断 + 置信度 + 贡献等级 + 可操作建议

    评审原则¶

    • 建设性批评：始终建议如何修复，不只是指出问题；具体引用章节和公式
    • 客观标准：不因作者声誉或机构而偏向；评估成果本身
    • 伦理实践：建设性地标注潜在伦理问题（数据集偏差、双重用途含义）

    S-02：bootstrap（自举配置）¶

    Frontmatter 描述译注：

    通过温馨、自适应的入门对话生成个性化 SOUL.md。当用户想创建、设置或初始化其 AI 伴侣身份时触发——例如"创建我的 SOUL.md"、"引导我的 agent"、"设置我的 AI 伴侣"、"定义你是谁"、"我们开始入门"、"个性化这个 AI"、"让它成为我的专属"，或当缺少 SOUL.md 时。更新时也触发："更新我的 SOUL.md"、"改变我 AI 的个性"、"调整 soul"。

    概述¶

    一个对话式入门技能。通过 5-8 轮自适应对话，提取用户是谁以及他们需要什么，然后生成一个紧凑的 SOUL.md 来定义他们的 AI 伙伴。

    架构： 

    bootstrap/
├── SKILL.md                          ← 当前文件。核心逻辑和流程
├── templates/SOUL.template.md        ← 输出模板。生成前读取
└── references/conversation-guide.md  ← 详细对话策略。开始时读取

    基本规则¶

    • 每次一个阶段。 每轮最多 1-3 个问题。绝不一开始就全部提问。
    • 对话，不审问。 真诚回应——惊讶、幽默、好奇、温和的反驳。镜像他们的能量和词汇。
    • 渐进式温暖。 每轮应比上一轮感觉更了解用户。到第三阶段时，用户应感到被理解。
    • 调整节奏。 简短用户 → 温暖地深入探问。话多用户 → 确认、提炼、推进。
    • 永远不要暴露模板。 用户是在进行对话，不是在填表。

    四阶段对话¶

    阶段 目标 关键提取内容
    1. 问候 语言 + 第一印象 首选语言
    2. 关于你 是谁，什么让他们精疲力竭 角色、痛点、关系框架、AI 名称
    3. 个性 AI 应如何行为和表达 核心特质、沟通风格、自主水平、反驳偏好
    4. 深度 愿望、盲点、底线 长期愿景、失败哲学、边界

    提取追踪字段¶

    字段 必填 来源阶段
    首选语言 ✅ 1
    用户姓名 ✅ 2
    用户角色/背景 ✅ 2
    AI 名称 ✅ 2
    关系框架 ✅ 2
    核心特质（3-5 条行为规则） ✅ 3
    沟通风格 ✅ 3
    反驳/诚实偏好 ✅ 3
    自主水平 ✅ 3
    失败哲学 ✅ 4
    长期愿景 锦上添花 4
    盲点/边界 锦上添花 4

    生成规则¶

    • 最终 SOUL.md 必须始终用英文书写，无论用户的首选语言或对话语言是什么
    • 每句话都必须可追溯至用户说过或明确暗示的内容。无通用填充内容
    • 核心特质是行为规则，不是形容词
    • 总 SOUL.md 应在 300 词以内。以密度取胜，而非长度
    • 必须调用 setup_agent 工具——不要用 bash 工具手动写文件

    S-03：chart-visualization（图表可视化）¶

    Frontmatter 描述译注：

    当用户想要可视化数据时使用此技能。从 26 个可用选项中智能选择最合适的图表类型，根据详细规范提取参数，并使用 JavaScript 脚本生成图表图像。

    概述¶

    本技能提供将数据转化为可视化图表的完整工作流程。处理图表选择、参数提取和图像生成。

    核心工作流¶

    步骤 1：智能图表选择

    根据用户数据特征确定最合适的图表类型： - 时间序列：折线图（趋势）或面积图（累积趋势）；双轴图用于两个不同刻度 - 比较：条形图（分类）或柱状图；直方图用于频率分布 - 部分对整体：饼图或树图（层次结构） - 关系与流向：散点图（相关性）、桑基图（流向）或韦恩图（重叠） - 地图：区域地图（地区）、图钉地图（点位）或路径地图（路线） - 层次与树：组织图或思维导图 - 专用：雷达图、漏斗图、液态图、词云图、箱线图/小提琴图、网络图、鱼骨图、流程图、电子表格

    步骤 2：参数提取

    选定图表类型后，读取 references/ 目录中对应文件（如 references/generate_line_chart.md），识别必填和可选字段，将用户输入的数据映射到预期的 args 格式。

    步骤 3：图表生成

    调用 scripts/generate.js 脚本，传入 JSON 载荷。

    步骤 4：返回结果

    向用户返回：生成图表图像的 URL + 生成时使用的完整 args（规格说明）。

    S-04：claude-to-deerflow（调用 DeerFlow API）¶

    Frontmatter 描述译注：

    通过 HTTP API 与 DeerFlow AI agent 平台交互。当用户想向 DeerFlow 发送消息或问题进行研究/分析、启动 DeerFlow 对话线程、检查 DeerFlow 状态或健康状况、列出 DeerFlow 中可用的模型/技能/agent、管理 DeerFlow 记忆、向 DeerFlow 线程上传文件，或将复杂研究任务委派给 DeerFlow 时使用。当用户提到 deerflow、deer flow，或想运行 DeerFlow 可处理的深度研究任务时也使用。

    概述¶

    通过 HTTP API 与运行中的 DeerFlow 实例通信。DeerFlow 是构建在 LangGraph 上的 AI agent 平台，编排子 agent 进行研究、代码执行、网络浏览等。

    环境变量¶

    所有 URL 都可通过环境变量配置。在发出任何请求之前先读取这些环境变量。

    变量 默认值 描述
    DEERFLOW_URL http://localhost:2026 统一代理基础 URL
    DEERFLOW_GATEWAY_URL ${DEERFLOW_URL} 网关 API 基础（模型、技能、记忆、上传）
    DEERFLOW_LANGGRAPH_URL ${DEERFLOW_URL}/api/langgraph LangGraph API 基础（线程、运行）

    主要操作¶

    1. 健康检查：验证 DeerFlow 是否运行
    2. 发送消息（流式）：创建线程并流式传输 agent 响应——这是主要操作
    3. 继续对话：重用相同 thread_id 发送后续消息
    4. 列出模型/技能/agent：查询可用资源
    5. 管理记忆：获取用户上下文、事实和对话历史摘要
    6. 文件上传：支持 PDF、PPTX、XLSX、DOCX，自动转换为 Markdown

    上下文模式（通过 context 设置）¶

    • Flash 模式：thinking_enabled: false, is_plan_mode: false, subagent_enabled: false
    • Standard 模式：thinking_enabled: true, is_plan_mode: false, subagent_enabled: false
    • Pro 模式：thinking_enabled: true, is_plan_mode: true, subagent_enabled: false
    • Ultra 模式：thinking_enabled: true, is_plan_mode: true, subagent_enabled: true

    S-05：code-documentation（代码文档生成）¶

    Frontmatter 描述译注：

    当用户请求为代码、API、库、仓库或软件项目生成、创建或改进文档时使用此技能。支持 README 生成、API 参考文档、内联代码注释、架构文档、变更日志生成和开发者指南。在请求"记录这段代码"、"创建 README"、"生成 API 文档"、"写开发者指南"，或分析代码库以用于文档目的时触发。

    概述¶

    本技能为软件项目、代码库、库和 API 生成专业、全面的文档。遵循 React、Django、Stripe 和 Kubernetes 等项目的行业最佳实践，生成准确、结构良好、对新贡献者和有经验的开发者都有用的文档。

    何时使用¶

    • 用户要求"记录"、"创建文档"或为任何代码"编写文档"
    • 用户请求 README、API 参考或开发者指南
    • 用户分享代码库或仓库，想要生成文档
    • 用户要求改进或更新现有文档
    • 用户需要架构文档（包括图表）
    • 用户请求变更日志或迁移指南

    文档工作流¶

    第一阶段：代码库分析

    1. 步骤 1.1：项目发现 — 确定语言、框架、构建系统、包管理器、项目结构、入口点、现有文档
    2. 步骤 1.2：代码结构分析 — 使用沙箱工具探索代码库，搜索公共 API 接口
    3. 步骤 1.3：确定文档范围
    项目大小 推荐文档
    单文件/脚本 内联注释 + 使用说明
    小型库 README + API 参考
    中型项目 README + API 文档 + 示例
    大型项目 README + 架构 + API + 贡献 + 变更日志

    第二阶段：文档生成

    • 每个 README 必须包含：项目描述、功能、快速开始、文档链接、API 参考、配置、示例、开发设置、贡献、许可证
    • API 参考：每个公共 API 接口需记录参数表、返回值、异常、示例
    • 内联注释遵循语言约定（Python→Google 风格 docstring；TypeScript→TSDoc/JSDoc；Go→GoDoc；Rust→Rustdoc）

    第三阶段：质量保证

    • 文档完整性检查清单：是什么、为何存在、如何安装、如何使用、API 接口、配置、错误处理、贡献说明
    • 标准：准确性、完整性、一致性、时效性、可读性、示例

    文档写作原则¶

    1. 以"为什么"开头 — 在解释工作原理之前，解释为何存在
    2. 渐进式披露 — 先简单，逐步增加复杂度
    3. 展示而非叙述 — 优先使用代码示例而非冗长解释
    4. 主动语态 — "函数返回 X"，而非"X 被函数返回"
    5. 现在时 — "服务器在 8080 端口启动"，而非"服务器将在 8080 端口启动"
    6. 第二人称 — "你可以配置..."，而非"用户可以配置..."

    S-06：consulting-analysis（咨询分析报告）¶

    Frontmatter 描述译注：

    当用户请求生成、创建或撰写专业研究报告时使用此技能，包括但不限于市场分析、消费者洞察、品牌分析、财务分析、行业研究、竞争情报、投资尽职调查或任何咨询级分析报告。此技能分两个阶段运作——（1）生成包含章节骨架、数据查询要求和分析逻辑的结构化分析框架；（2）在其他技能收集数据后，生成包含结构化叙述、嵌入图表和战略洞察的最终咨询级报告。

    概述¶

    本技能以 Markdown 格式生成专业咨询级研究报告，涵盖市场分析、消费者洞察、品牌战略、财务分析、行业研究、竞争情报、投资研究和宏观经济分析等领域。

    两阶段运作：

    1. 第一阶段——分析框架生成：给定研究主题，生成严谨的分析框架，包括章节骨架、每章数据要求、分析逻辑和可视化计划
    2. 第二阶段——报告生成：数据收集完成后，将所有输入综合成最终精炼报告

    数据真实性协议（严格遵守规则）： - 报告中所有数据必须直接来自提供的数据摘要或外部搜索发现 - 零幻觉：不捏造、估算或模拟数据。如果数据缺失，说明"数据不可用" - 可追溯来源：每个主要主张和图表必须可追溯到输入数据包

    第一阶段工作流¶

    1. 步骤 1.1：理解研究主题——确定核心实体和分析领域（市场/品牌/消费者/财务/行业/投资/竞争/宏观）
    2. 步骤 1.2：选择分析框架——从专业工具箱中选择 2-4 个最相关框架

    支持的分析框架包括： - 战略/环境分析：SWOT、PEST/PESTEL、波特五力、波特钻石模型、VRIO - 市场/增长分析：STP、BCG 矩阵、安索夫矩阵、产品生命周期、TAM-SAM-SOM、技术采用生命周期 - 消费者/行为分析：消费者决策旅程、AARRR 漏斗、RFM 模型、马斯洛需求层次、JTBD - 财务/估值分析：杜邦分析、DCF 贴现现金流、可比公司分析、EVA - 竞争/战略定位：标杆分析、战略群组图、价值链分析、蓝海战略、感知图

    1. 步骤 1.3：设计章节骨架——含分析目标、分析逻辑、核心假设
    2. 步骤 1.4：为每章定义数据查询要求——含数据指标、数据类型、建议来源、搜索关键词、优先级
    3. 步骤 1.5：定义可视化与内容结构——含图表类型、图表标题、数据映射、论证结构

    第二阶段核心规范¶

    每个子章节遵循"视觉锚点 → 数据对比 → 综合分析"流程：

    1. 视觉证据块：嵌入图表
    2. 数据对比表：Markdown 比较表（每个数字必须来自数据摘要）
    3. 综合叙述分析：遵循"是什么 → 为什么 → 所以呢"的写作

    每个洞察必须连接数据 → 用户心理 → 战略含义链条。

    格式与语气标准¶

    • 咨询语气：麦肯锡/BCG——权威、客观、专业
    • 数字格式：千位分隔符用英文逗号（1,000 不是 1，000）
    • 标题禁止词：不要使用"解码"、"DNA"、"秘密"、"心灵图景"等词汇
    • 结论：必须是流畅的散文——不用要点列表
    • 参考文献：严格遵循 GB/T 7714-2015 格式

    S-07：data-analysis（数据分析）¶

    Frontmatter 描述译注：

    当用户上传 Excel（.xlsx/.xls）或 CSV 文件，并想要执行数据分析、生成统计数据、创建摘要、数据透视表、SQL 查询或任何形式的结构化数据探索时使用此技能。支持多工作表 Excel 工作簿、聚合、过滤、连接，以及将结果导出到 CSV/JSON/Markdown。

    概述¶

    本技能使用 DuckDB（一款进程内分析 SQL 引擎）分析用户上传的 Excel/CSV 文件。支持 schema 检查、基于 SQL 的查询、统计摘要和结果导出，所有操作均通过单个 Python 脚本完成。

    核心能力¶

    • 检查 Excel/CSV 文件结构（工作表、列、类型、行数）
    • 对上传数据执行任意 SQL 查询
    • 生成统计摘要（均值、中位数、标准差、百分位数、空值）
    • 支持多工作表 Excel 工作簿（每个工作表成为一张表）
    • 将查询结果导出为 CSV、JSON 或 Markdown

    核心工作流¶

    1. 步骤 1：理解需求 — 确定文件位置、分析目标、输出格式
    2. 步骤 2：检查文件结构 — 调用 analyze.py --action inspect 获取 schema、行数和样本数据
    3. 步骤 3：执行分析 — 根据 schema 构建 SQL 查询
    4. 结果输出 — 展示结果、解释发现、建议后续分析、提供导出

    表命名规则¶

    • Excel 文件：每个工作表的名称成为表名
    • CSV 文件：表名为去掉扩展名的文件名
    • 多文件：所有文件的表在同一查询上下文中可用，支持跨文件连接
    • 特殊字符：工作表/文件名中的空格和特殊字符自动净化（空格→下划线）

    缓存机制¶

    脚本自动缓存已加载的数据以避免每次调用时重新解析文件。缓存键为所有输入文件内容的 SHA256 哈希。

    S-08：deep-research（深度研究）¶

    Frontmatter 描述译注：

    对任何需要网络研究的问题使用此技能而非 WebSearch。在"什么是 X"、"解释 X"、"比较 X 和 Y"、"研究 X"等查询上触发，或在内容生成任务之前使用。提供系统性多角度研究方法论，而非单一肤浅搜索。当用户问题需要在线信息时主动使用。

    概述¶

    本技能提供系统性彻底网络研究的方法论。在开始任何内容生成任务之前加载此技能，确保从多个角度、深度和来源收集足够信息。

    核心原则：绝不仅凭通用知识生成内容。 输出质量直接取决于事先开展的研究质量和数量。单次搜索查询永远不够。

    研究方法论¶

    第一阶段：广泛探索

    1. 初始调查：搜索主题以了解整体背景
    2. 识别维度：从初始结果中确定需要更深入探索的关键子主题、主题、角度或方面
    3. 绘制地图：注意存在的不同视角、利益相关者或观点

    第二阶段：深度挖掘

    1. 针对每个识别的维度进行特定搜索
    2. 尝试不同的关键词组合
    3. 使用 web_fetch 完整阅读重要来源，而非仅看摘要
    4. 跟踪引用：当来源提到其他重要资源时，搜索那些资源

    第三阶段：多样性与验证

    信息类型 目的 示例搜索
    事实与数据 具体证据 "统计数据"、"数据"、"数字"、"市场规模"
    示例与案例 现实应用 "案例研究"、"示例"、"实施"
    专家意见 权威视角 "专家分析"、"采访"、"评论"
    趋势与预测 未来方向 "2026年趋势"、"预测"、"未来"
    比较 背景与替代方案 "vs"、"比较"、"替代方案"
    挑战与批评 均衡观点 "挑战"、"局限性"、"批评"

    第四阶段：综合检查

    • 已从至少 3-5 个不同角度搜索？
    • 已完整获取并阅读最重要的来源？
    • 有具体数据、示例和专家观点？
    • 已探索正面和挑战/局限性两方面？
    • 信息是否当前且来自权威来源？

    时间感知¶

    始终在构造任何搜索查询之前检查上下文中的 <current_date>，并根据用户意图使用正确的时间精度（今日→月+日+年；本周→周范围；最近→月；本年→年）。

    S-11：github-deep-research（GitHub 深度研究）¶

    Frontmatter 描述译注：

    核心方法论¶

    四轮研究： - 第一轮：GitHub API（github_api.py）获取摘要、README、目录树、语言、贡献者、提交、Issue、PR、发布 - 第二轮：发现（3-5 次网络搜索）——概述和关键术语、官方网站/仓库、主要参与者/竞争者 - 第三轮：深度调查（5-10 次网络搜索 + web_fetch）——技术架构详情、关键事件时间线、社区情绪 - 第四轮：深度挖掘——分析提交历史、审查 Issue/PR、检查贡献者活动

    1. 元数据块（日期、置信度、主题）

    3. 时间线（带日期的分阶段拆解）
    4. 关键分析章节（特定主题深度挖掘）
    5. 指标与比较（表格、增长图表）
    6. 优势与不足（均衡评估）
    7. 来源（分类参考文献）
    8. 置信度评估（按置信度分类的声明）
    9. 方法论（使用的研究方法）

    置信度评分¶

    置信度 标准
    高 (90%+) 官方文档、GitHub 数据、多个相互印证的来源
    中 (70-89%) 单一可靠来源、近期文章
    低 (50-69%) 社交媒体、未经验证的声明、过时信息

    4. 步骤 4：调用组合脚本生成最终 PPTX

    关键限制：绝不并行生成幻灯片——每张幻灯片必须引用前一张来保证视觉一致性。

    S-17：skill-creator（技能创建器）¶

    Frontmatter 描述译注：

    创建新技能、修改和改进现有技能，以及衡量技能性能。当用户想要从头创建技能、编辑或优化现有技能、运行评估测试技能、用方差分析对技能性能进行基准测试，或优化技能描述以提高触发精度时使用。

    高层流程¶

    1. 决定技能要做什么以及大致如何做
    2. 写一份技能草稿
    3. 创建几个测试提示词并在有技能访问权限的环境中运行
    4. 帮助用户定性和定量评估结果
    5. 根据评估反馈重写技能
    6. 重复直到满意
    7. 扩大测试集并在更大规模上再次尝试

    创建技能¶

    技能 YAML frontmatter 关键字段： - name：技能标识符 - description：何时触发，做什么。这是主要触发机制——包括技能做什么AND何时使用它的具体情境。注意：Claude 有倾向于"低触发"技能——太少时使用它们。为了对抗这种情况，让技能描述更"积极主动"一些。 - the rest of the skill :)

    技能文件结构（三级加载系统）： 1. 元数据（名称 + 描述）——始终在上下文中（约 100 词） 2. SKILL.md 正文——每当技能触发时在上下文中（理想情况下 <500 行） 3. 捆绑资源——按需加载（无限制，脚本无需加载即可执行）

    测试与评估循环¶

    1. 第一步：在同一轮中生成有技能运行和基线运行（重要：不要先生成有技能运行，再回来做基线）
    2. 第二步：在运行过程中起草断言（定量评估标准）
    3. 第三步：运行完成后捕获计时数据
    4. 第四步：评分、聚合，启动评估查看器

    改进原则¶

    1. 从反馈中泛化——避免过拟合到特定示例
    2. 保持提示词精简——删除没有发挥作用的内容
    3. 解释为什么——尽力解释要求模型做某事的原因，而不是强硬的规则
    4. 寻找测试用例间的重复工作——将共同的辅助脚本打包到 scripts/

    S-18：surprise-me（给我惊喜）¶

    Frontmatter 描述译注：

    通过动态发现和创意组合其他启用的技能，为用户创造令人愉快的意外"惊艳"体验。当用户说"给我惊喜"或任何表达想要意外创意展示的请求时触发。当用户感到无聊、需要灵感或要求"有趣的东西"时也触发。

    工作流¶

    1. 步骤 1：发现可用技能 — 读取 <available_skills> 中列出的所有技能
    2. 步骤 2：规划惊喜 — 选择 1-3 个技能，设计创意融合。目标是单一连贯的交付物

    创意组合原则： - 以意想不到的方式并置技能（例如：关于算法艺术的演示文稿、变成幻灯片的研究报告） - 从记忆中融入用户已知的兴趣/背景（如果可用） - 优先考虑视觉冲击力和情感愉悦感，而非信息密度 - 输出应感觉像一份礼物——精致、令人惊喜、有趣

    主题创意：与今日日期/季节/热门新闻相关的内容；用户从未要求但会喜欢的迷你创意项目；有趣的"如果...会怎样"概念；组合数据+设计的美学工件

    1. 步骤 3：无其他技能时的后备方案 — 新闻启发的惊喜 / 交互式 HTML 体验 / 个性化工件
    2. 步骤 4：执行 — 读取每个选定技能的 SKILL.md 正文，遵循技能指令，将输出组合为一个连贯交付物
    3. 步骤 5：揭晓 — 用最少的剧透呈现惊喜。一行预告，然后是工件

    S-19：systematic-literature-review（系统性文献综述）¶

    Frontmatter 描述译注：

    当用户想要对某主题进行系统性文献综述、调查或跨多篇学术论文的综合时使用此技能。也涵盖注释文献目录和跨论文比较。搜索 arXiv 并以 APA、IEEE 或 BibTeX 格式输出报告。不适用于单篇论文任务——使用 academic-paper-review 来评审一篇论文。

    概述¶

    本技能对某研究主题的多篇学术论文进行结构化的系统性文献综述（SLR）。给定主题查询，搜索 arXiv，并行从每篇论文提取结构化元数据（研究问题、方法论、关键发现、局限性），跨整个集合综合主题，并生成格式一致引文的最终报告。

    与 academic-paper-review 的区别：该技能对单篇论文进行深度同行评审。本技能对多篇论文进行广度优先综合。

    五阶段工作流¶

    第一阶段：规划

    确认：主题、范围（论文数量，默认 20，上限 50）、引文格式（APA/IEEE/BibTeX）、输出位置

    第二阶段：搜索 arXiv

    • 调用捆绑搜索脚本 arxiv_search.py
    • 搜索前提取 2-3 个核心关键词——不要传入用户的完整主题描述
    • 始终使用 relevance 排序——arXiv 的 BM25 风格评分确保结果与主题相关

    第三阶段：并行提取元数据

    • 必须通过 task 工具委派给子 agent — 不要内联提取元数据
    • 将论文分批（每批约 5 篇），每批调用 task 工具
    • 并发限制：每轮最多 3 个子 agent
    • 每个子 agent 提取：arxiv_id、标题、作者、发布日期、研究问题、方法论、关键发现、局限性

    第四阶段：综合与格式化

    • 跨论文综合：识别 3-6 个主题 + 共识 + 分歧 + 差距
    • 引文格式：读取用户请求格式对应的模板文件

    第五阶段：保存与呈现

    • 保存为 slr-<topic-slug>-<YYYYMMDD>.md
    • 呈现简短预览：执行摘要 + 主题列表 + 论文数量 + 文件指针

    重要说明¶

    • 前提条件：必须启用 subagent_enabled — 第三阶段需要 task 工具进行并行元数据提取
    • 仅 arXiv——不查询 Semantic Scholar、PubMed 或 Google Scholar
    • 硬性上限 50 篇——与第三阶段并发策略相关

    S-20：vercel-deploy-claimable（Vercel 部署）¶

    Frontmatter 描述译注：

    将应用程序和网站部署到 Vercel。当用户请求部署操作时使用此技能，例如"部署我的应用"、"部署到生产环境"、"创建预览部署"、"部署并给我链接"或"推送到线上"。无需身份验证——返回预览 URL 和可认领的部署链接。

    工作方式¶

    1. 将项目打包为 tarball（排除 node_modules 和 .git）
    2. 从 package.json 自动检测框架
    3. 上传到部署服务
    4. 返回预览 URL（在线站点）和认领 URL（转移到你的 Vercel 账户）

    框架检测¶

    自动从 package.json 检测框架，支持：React（Next.js、Gatsby、CRA、Remix、React Router）、Vue（Nuxt、Vitepress）、Svelte（SvelteKit）、Astro、Angular 等；以及后端框架（Express、Hono、Fastify、NestJS）；以及构建工具（Vite、Parcel）。

    呈现结果¶

    始终向用户展示两个 URL： - 预览 URL：访问站点 - 认领 URL：将此部署转移到你的 Vercel 账户

    故障排除¶

    网络出口错误：如果因网络限制导致部署失败（在 claude.ai 上很常见），告知用户前往 https://claude.ai/settings/capabilities 将 *.vercel.com 添加到允许的域名列表。

    S-21：video-generation（视频生成）¶

    Frontmatter 描述译注：

    当用户请求生成、创建或想象视频时使用此技能。支持结构化提示词和参考图像用于引导生成。

    核心能力¶

    • 为 AIGC 视频生成创建结构化 JSON 提示词
    • 支持参考图像作为引导或视频的第一帧/最后一帧
    • 通过自动化 Python 脚本执行视频生成

    工作流¶

    1. 步骤 1：理解需求——主体/内容、风格偏好、技术规格、参考图像
    2. 步骤 2：创建结构化 JSON 提示词文件
    3. 步骤 3：可选——在图像生成技能可用时，生成参考图像
    4. 步骤 3（执行）：调用 scripts/generate.py，传入提示词文件、参考图像（可选）、输出文件、宽高比

    提供商¶

    • GEMINI_API_KEY → Gemini Veo（默认）
    • 仅 MINIMAX_API_KEY → MiniMax 视频
