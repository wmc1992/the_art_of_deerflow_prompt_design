    序言：为什么要读一个框架的 Prompt¶

    2026 年初，当字节跳动将 DeerFlow 的源代码公开发布时，在开源社区引发了一轮不小的讨论。开发者们快速 fork、部署、测试，关注点集中在它的架构设计——LangGraph 驱动的多 Agent 框架、沙箱隔离执行、MCP 协议集成。

    但有一个部分被忽视了：它的 Prompt 写得有多认真。

    一行 Prompt 背后的工程决策¶

    打开 backend/packages/harness/deerflow/agents/lead_agent/prompt.py，你会看到一个超过 200 行的系统 Prompt 模板。这不是一个随手写就的字符串拼接，它是一个经过精心设计的工程构件，每一个段落都有明确的功能职责，每一行规则都背后都有深思熟虑的取舍。

    以澄清系统（P-01 中的 <clarification_system> 段落）为例。它没有简单写"遇到不清楚的地方要问用户"，而是：

    • 定义了 5 种具名场景（missing_info、ambiguous_requirement、approach_choice、risk_confirmation、suggestion），把抽象的"不清楚"具体化为可识别的类型
    • 明确了工作流优先级：澄清 → 计划 → 行动，不允许颠倒顺序
    • 对每个场景都给出了具体示例，消除模型对规则的多义理解
    • 用 ❌/✅ 格式的对比列表强化禁止行为，比单纯的"不要"规则有力得多

    这不只是在写 Prompt，这是在设计约束系统。

    研究框架 Prompt 的价值¶

    你也许会问：我自己项目的场景和 DeerFlow 完全不同，研究它的 Prompt 有什么意义？

    这个问题的答案在于：设计思想是可迁移的，具体 Prompt 才是场景特定的。

    DeerFlow 的 Prompt 体系集中展示了工业级 Agent 系统中几个最难处理的问题是如何被解决的：

    问题一：系统 Prompt 越来越长，怎么控制代价？

    DeerFlow 的回答是"前缀缓存优先（Prefix Cache First）"原则：系统 Prompt 保持完全静态，不嵌入任何随用户/会话变化的内容（记忆、日期、用户名）。动态内容统一通过 DynamicContextMiddleware 以 HumanMessage 的形式注入，完全不触碰系统 Prompt。这样，无论多少用户并发，LLM 提供商的前缀缓存始终命中，静态 Prompt 部分的计算成本接近于零。

    问题二：如何防止模型在复杂任务中"迷失"？

    DeerFlow 的回答是 XML 标签锚定结构。<role>、<thinking_style>、<clarification_system>、<skill_system>、<working_directory>、<response_style>、<citations>、<critical_reminders> 八个标签将 Prompt 切割为功能单元，每个单元都有独立的语义边界。这让模型可以"按需激活"相关段落，而不是把 200 行文本当作一个扁平的规则列表。

    问题三：工具文档（docstring）算不算 Prompt？

    算，而且非常重要。DeerFlow 的工具 docstring（P-21 至 P-25）都有一个共同特征：明确说明何时用、何时不用。ask_clarification 的描述花了大量篇幅告诉模型哪五种场景必须用它，而不是只说它能做什么。这种"使用场景说明优先于功能描述"的写法，是降低工具被误用概率的关键。

    问题四：如何在运行时阻止模型进入无限循环？

    DeerFlow 的 Loop 检测中间件（P-13 至 P-16）用四条消息构成了一套分级干预机制：警告 → 频率警告 → 强制停止 → 频率强制停止。关键的设计细节是：这些消息被注入为 HumanMessage，而非 AIMessage，这样不会破坏消息序列的工具调用配对完整性。这种"干预不破坏协议"的设计思路，在工业系统中随处可见的价值。

    本书的立场¶

    本书不是 DeerFlow 的使用手册，也不是 LangGraph 的教程。本书的立场是：把 DeerFlow 的 Prompt 体系当作一个工程案例，从中提炼设计原则。

    每一章都会给出原始 Prompt（英文原文，与源码一字不差），然后逐步解构它的设计选择——为什么这样写、写这样有什么效果、如果换一种写法会发生什么。

    这种分析方式需要你对大语言模型的基本行为有一定了解——知道 token budget 是什么、为什么 few-shot 示例有效、注意力机制大体如何工作。如果你是算法工程师，这些背景知识你大概率已经具备。

    如果你只是想快速获取所有 Prompt 原文，可以直接跳到附录 A（英文原文）和附录 B（中文译注）。这两个附录是自洽的参考资料，不依赖正文。

    一个阅读前的承诺¶

    本书对原始 Prompt 文本保持了绝对的忠实。附录 A 中的所有英文 Prompt 与 DeerFlow v2.0.0-rc1 源码中的对应内容完全一致，经过逐字核对。本书完成后会专门进行一次校验。任何分析或解读，都不会修改原文，只会在原文之后标注。

    如果你在阅读过程中发现原文与源码有任何出入，这是一个需要被指出的错误，请告知作者。

    本书基于 DeerFlow v2.0.0-rc1（GitHub tag 打出时间：2026-06-19）。
    本书涵盖的 Prompt 文件路径均相对于仓库根目录 backend/packages/harness/。
